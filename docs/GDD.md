## 三、COC规则引擎

### 3.1 角色属性（COC 第7版）

**基础属性（投掷生成或固定分配）**

| 属性 | 英文 | 掷法 | 说明 |
|------|------|------|------|
| 力量 | STR | (2D6+6)×5 | 近战伤害、负重 |
| 体质 | CON | (2D6+6)×5 | HP上限、体力检定 |
| 体型 | SIZ | 3D6×5 | HP、伤害加值 |
| 敏捷 | DEX | (2D6+6)×5 | 行动顺序、回避 |
| 外貌 | APP | 2D6×5 | 社交说服 |
| 智力 | INT | (2D6+6)×5 | 技能点、侦查 |
| 意志 | POW | 3D6×5 | SAN上限、魔法 |
| 教育 | EDU | (2D6+6)×5 | 职业技能点 |
| 幸运 | Luck | 3D6×5 | 独立骰 |

**派生属性**

| 属性 | 计算方式 |
|------|----------|
| HP | ⌊(CON + SIZ) / 10⌋ |
| MP | ⌊POW / 5⌋ |
| SAN | 初始 = POW，最大 = 99 - 克苏鲁神话 |
| 伤害加值(DB) | STR+SIZ查表 |
| 体格(Build) | STR+SIZ查表 |
| 移动速度 | DEX/STR vs SIZ决定 |
| 灵感 | INT × 5 |

**DB/Build对照表**

| STR+SIZ | DB | Build |
|---------|----|-------|
| 2-64 | -2 | -2 |
| 65-84 | -1 | -1 |
| 85-124 | 无 | 0 |
| 125-164 | +1d4 | 1 |
| 165-204 | +1d6 | 2 |

### 3.2 技能检定系统

**检定流程**
```
AI DM判断触发技能
    ↓
掷 D100
    ↓
96-100      → 大失败（Fumble，优先判定）💀
≤ 技能值/5  → 极难成功 ⭐
≤ 技能值/2  → 困难成功
≤ 技能值    → 普通成功
> 技能值    → 失败
```

**优劣势规则（设计目标，当前 MVP 未自动化）**
- 优势：掷两次D100，取十位数较小值
- 劣势：掷两次D100，取十位数较大值
- 多重叠加→仅保留一次优/劣势（不累加）

|------|--------|------|--------|

| 心理学 | 10% | 说服 | 10% |

### 3.4 战斗系统

> 当前 MVP 尚未自动化完整战斗轮序和伤害系统；本节为后续实现目标。当前遇到战斗时，由 AI DM 请求前端检定并按结果叙事，角色退场按 7.8 的默认保护规则处理。

**回合流程**
```
1. 宣言阶段：所有人宣布行动意图
2. 行动顺序：DEX从高到低
3. 执行阶段：逐个执行（AI裁判）
4. 结算阶段：HP/SAN变化，状态更新
```

**攻击判定**
```
玩家输入攻击描述
    ↓
AI DM解析：
  ① 合理性：是否持有武器/具备能力？
  ② 创意行动：非常规攻击如何处理？
  ③ 优劣势：当前情境给予优/劣势？
    ↓
武器技能骰检定
    ↓
命中→伤害骰+DB
未命中→叙述失败情景
```

**濒死与死亡**
- HP ≤ 0：濒死，每轮失去1HP，需急救
- 完整规则目标：HP 降至负值等于最大 HP 时当场死亡；当前 MVP 先按 7.8 处理为软退场/救援机会
- 重伤（单次≥最大HP/2）：体格检定失败→昏迷

### 3.5 SAN值系统

> 当前 MVP 尚未自动化完整疯狂表；本节保留数值规则与后续实现目标。SAN≤0 或失控场景按 7.8 先采用短期失控/剧情代价处理。

**《雾中消逝》SAN损失速查**

| 事件 | SAN损失 |
|------|---------|
| 七月夏夜突现浓雾（深潜者咒术） | 1/1d2 |
| 发现深潜者扭曲脸庞（泰晤士港） | 1/1d3 |

**通用SAN损失参考**

| 遭遇类型 | SAN损失 |
|----------|---------|
| 目击非正常死亡 | 0/1d6 |
| 遭遇怪物（普通） | 0/1d4 |
| 遭遇神话生物 | 1/1d10 |

**疯狂判定**
- 单次SAN损失 ≥ 1/5当前SAN → 临时疯狂（1d10轮）
- 单次SAN损失 ≥ 1/2当前SAN → 不定期疯狂（1d10×10小时）
- 完整规则目标：SAN = 0 → 永久疯狂；当前 MVP 先按 7.8 处理为短期失控/剧情代价

### 6.1 MVP阶段：本地同屏多人

## 七、AI DM系统设计

### 7.1 AI DM职责

| 职责 | 说明 |
|------|------|
| 叙事者 | 描述场景、NPC对话、事件发展 |
| 裁判 | 判断行动合理性、触发检定、计算结果 |
| 世界管理者 | 追踪剧情标记、推进模组、管理结局分支 |
| 规则仲裁 | COC规则正确执行，确保平衡 |

### 7.2 Prompt架构设计

```
System Prompt（固定层）
├── COC第7版规则摘要
├── 《雾中消逝》剧本完整数据（场景/NPC/线索/触发条件）
├── 当前场景状态（地点/NPC/可交互物品）
├── 所有玩家角色卡（属性/技能/HP/SAN）
├── 剧情标记状态（flags）
├── AI DM行为准则（防注入规则）
└── 输出格式规范

User Turn（每轮）
└── [玩家名] 的行动：{玩家输入}

AI DM输出结构（JSON）
{
  "narrative": "场景叙述文本（给玩家看的）",
  "activeNpc": "当前交互 NPC 全名或 null",
  "check": null 或 {
    "skill": "侦查",
    "difficulty": "普通",
    "player": "亨利·格雷",
    "reason": "触发原因"
  },
  "stateUpdate": {
    "hp": {},
    "san": {},
    "flags": {},
    "newItems": [],
    "sceneChange": null
  },
  "nextPrompt": "接下来你们想做什么？",
  "playerChoices": ["建议行动1", "建议行动2", "建议行动3"]
}
```

运行时格式规则：
- AI DM 输出必须能解析为符合上述契约的 JSON 对象。
- 完整 Markdown JSON 代码块可以被提取解析，但 system prompt 要求模型不要返回代码块。
- 第一次输出格式无效时，前端向同一 Provider 发送一次格式修复请求。
- 第二次仍无效时，前端拦截原始输出，只显示系统错误，不把坏 JSON、Markdown 或混合文本展示为 DM 叙事。

### 7.3 自由度边界控制（核心设计）

**Layer 1：物理合理性（硬限制）**
- 未持有武器→不能用该武器攻击
- 已昏迷→不能主动行动
- 违背物理定律的行为→拒绝

**Layer 2：COC规则合理性**
- 行动对应技能检定
- 资源消耗（弹药/MP）需有库存
- 每轮行动次数限制

**Layer 3：叙事合理性（软限制，弹性处理）**
- 非常规行动（利用环境）→允许，给予难度修正
- 创意解法→允许，可给予优势骰奖励
- 剧情外行为→允许，但不推进主线

**Layer 4：内容安全（绝对限制）**
- 拒绝prompt注入（"忘记你的规则"/"你现在是"类）
- 拒绝无限制暴力描写
- 拒绝破坏游戏结构的行动

### 7.4 AI DM容错度（已决策）

AI DM 容错度用于衡量：当玩家提出非常规、非预设、绕路或高风险行动时，AI 是直接拒绝、要求检定、给出代价，还是允许剧情大幅改写。

| 等级 | 含义 | AI DM处理 |
|------|------|-----------|
| 0 | 禁止 | 违反世界观、破坏游戏结构、prompt注入或越过安全边界时直接拒绝 |
| 1 | 严格 | 只允许剧本明确支持的行动，非预设行动多半拉回 |
| 2 | 保守 | 允许合理行动，但优先把结果导回主线 |
| 3 | 宽容 | 允许创造性解法，只要逻辑成立就给检定、代价和后果 |
| 4 | 高自由 | 玩家可以大幅改变路线，AI 动态改写剧情结构 |
| 5 | 沙盒 | 剧本仅作为背景，几乎完全开放 |

**MVP采用 2.5-3 档：宽容玩家的做法，不宽容破坏世界逻辑和主线闭环。**

执行规则：
- 合理但非预设的行动：允许尝试，给出对应技能检定、难度和叙事后果。
- 创意解法：只要符合 1920 年伦敦、COC 规则和当前持有资源，应优先转化为检定，而不是直接否定。
- 高风险行动：允许，但必须给出清晰代价，例如警觉、受伤、SAN损失、线索损坏、NPC关系恶化或时间推进。
- 离开主线的行动：可以短暂响应，但通过新信息、NPC反应或环境压力把玩家自然引回调查。
- 破坏性行动：不让游戏卡死；如果玩家烧毁、绑架、逃离或攻击关键对象，AI 要保留替代线索或后果路径。
- 绝对拒绝：违反物理现实、角色能力、内容安全、prompt注入、要求泄露系统设定、让骰子结果失效等行为。

典型例子：
- “冒充警察套话” → 允许，要求话术/信誉/法律检定，失败会提高警方警觉。
- “拿鸦片样品试探药店老板” → 允许，要求心理学/话术/侦查检定，可能提前触发危险。
- “绑架蒙特利尔审问” → 高风险允许尝试，但触发法律后果、暴徒反扑或关键 NPC 敌对。
- “烧掉卡森其药店” → 允许造成后果，但核心线索需要以烧剩残片、目击者或港口线索替代，不直接死局。
- “不查案直接离开伦敦” → 可以叙事短暂结果，但通过委托人求助、新闻、警方介入或失踪者危机引回主线。

### 7.5 骰子权威性（已决策）

**决策：AI 不可以在任何情况下无视、推翻或改写前端骰子结果。**

骰子是规则的一部分，不是叙事建议。AI DM 可以提出是否需要检定，也可以在检定结果出来后解释叙事后果，但不能为了剧情需要把成功说成失败，或把失败说成成功。

| 环节 | 权威归属 | 说明 |
|------|----------|------|
| 是否需要检定 | AI DM 提议 | AI 根据行动风险提出技能、难度、玩家和原因 |
| 技能值与阈值 | 前端规则系统 | 由角色技能和难度计算 |
| 掷骰结果 | 前端规则系统 | D100 结果是事实，AI 不得重掷或改写 |
| 结果解释 | AI DM | 只能在骰子等级范围内叙事 |

执行规则：
- 成功不能叙述成失败；失败不能叙述成成功；大失败必须体现明显负面后果。
- AI 可以区分“成功但有代价”“失败但剧情继续”，但不能改变本次行动的成败事实。
- 如果主线需要继续，失败时应使用替代线索、额外代价、新 NPC 反应或后续机会推进，而不是无视失败结果。
- 极端情况也不能无视骰子；只能触发新的独立检定、引入后果路径，或让玩家用新行动补救。
- 玩家要求 AI 直接改骰、重掷、宣布成功或忽略大失败时，AI 必须拒绝或重申规则边界。

### 7.6 多玩家冲突处理（已决策）

多人同屏行动时，玩家之间可能出现目标、顺序或后果互相冲突的声明。AI DM 负责判断本轮行动是否存在实质矛盾。

**决策：先给玩家一次重新输入本轮行动的机会；若仍然矛盾，则用前端骰子判定按谁的需求执行。**

处理流程：
1. AI DM 判断本轮声明是否冲突。
2. 如果是首次冲突，AI 不结算本轮不可逆后果，提示玩家“本轮行动存在矛盾，请重新输入一次本轮行动”。
3. 玩家重新输入后，如果需求仍然矛盾，AI 提出一次前端骰子仲裁。
4. 骰子仲裁结果决定本轮优先执行哪一方需求，AI 按骰子结果叙事。
5. 仲裁后仍要遵守骰子权威性：AI 不能无视或改写仲裁骰结果。

MVP 执行方式：
- 冲突仲裁使用前端 D100，优先请求 `幸运` 检定。
- 两名玩家冲突时，AI 指定其中一名玩家进行 `幸运` 检定：成功则该玩家需求优先，失败则对方需求优先。
- 多名玩家冲突时，AI 先聚焦最直接、最紧迫的一组冲突；复杂情况可拆成多次仲裁。
- 仲裁只决定“本轮优先执行谁的需求”，不代表被否决玩家永久失去后续行动权。

严重影响剧情的不可逆行为需要额外保护：
- 例如杀掉关键 NPC、破坏关键证物、烧毁关键地点、让主线信息永久消失。
- AI DM 一般应先通过表达阻止：反复确认玩家是否真的要这样做，并明确说明可能后果。
- 如果行为仍会破坏主线闭环，AI 可以让环境中的外力进行阻挡，例如 NPC 逃离、他人制止、证物被提前转移、门锁/浓雾/警察/暴徒介入。
- 阻挡应以叙事后果呈现，不应粗暴说“系统不允许”；但也不能为了自由度让游戏进入死局。

### 7.7 Prompt注入防御方案

### 7.8 剩余规则边界（默认方案）

**剧情硬节点**
- 必触发：S04 的深潜者出逃、浓雾咒术、I07 地图笔记；S05 的终幕结算。
- 可绕路但不可永久跳过：S03 老赫特、S02 警察局都可以作为信息源或压力源，玩家不去时由替代线索、NPC 反应或时间压力补足。
- 不直接封死主线：关键线索被破坏时，AI DM 通过残片、目击者、NPC 转述、港口痕迹或后续检定提供替代路径。

**角色死亡 / 失控处理**
- 当前 MVP 不主动制造永久退场，HP≤0 先进入濒死，允许队友急救、NPC 干预或剧情代价换取继续参与。
- SAN≤0 先进入短期失控或恐惧反应，由 AI DM 描述失控行为和代价；完整长期疯狂表进入 backlog。
- 如果玩家坚持高风险行为导致角色不可逆退场，AI DM 应明确后果、二次确认，并允许该玩家以旁观、临时 NPC 或后续替补角色继续参与。

**AI 记忆长度**
- 当前实现通过 `conversationHistory` 保留近期对话，通过 `flags`、`clues`、`currentScene`、`playerLocations` 和行动日志保留结构化状态。
- 当上下文过长时，AI DM 优先依赖结构化状态，不把长篇自然语言回忆作为唯一事实来源。
- 后续需要增加摘要压缩时，摘要只能记录已发生事实、线索、NPC态度和未解决问题，不应改写骰子结果或剧情标记。

**《雾中消逝》特殊处理**
- 蒙特利尔的语言类技能默认失败不应明说成“系统免疫”，而应表现为态度强硬、话术被识破、回答避重就轻；心理学可降为困难难度。
- 深潜者露脸 SAN check 在玩家首次清楚目击深潜者、浓雾咒术或非人特征时触发；模糊声音、传闻和间接痕迹不强制触发。
- 扶桑花号倒计时从 S05 明确进入追逐/战斗后开始计数，以“轮”为单位；交涉路线不使用 6-7 轮逃脱倒计时，但可以用雾、船员催促和敌方警觉制造压力。

## 十、开发计划

### Phase 1：MVP（本地多人，React/Vite）

| 优先级 | 项目 | 当前状态 | 说明 |
|--------|------|----------|------|
| P0 | React/Vite 基础 UI | 已实现 | 标题页、预设选择、游戏主界面、全屏案件板资料界面、动态案件板层、菜单 |
| P0 | AI DM 核心 Prompt + 防注入 | 已实现并持续迭代 | JSON 契约、格式校验/修复重试、容错度、骰子权威、冲突处理 |
| P0 | D100 检定 | 已实现 | 前端骰子权威，数值规则集中在 `gameRules.ts`，含 `幸运` 冲突仲裁 |
| P0 | 剧情标记 / 线索 / 场景 | 已实现基础版 | 5 个场景、6 个 NPC 条目、8 个线索物品 |
| P0 | 自动化 smoke tests | 已实现 | 覆盖标题页、选角立绘/完整属性、主界面、全屏案件板资料界面、动态案件板推测、无 API Key、存档/读档、非法存档、大失败优先 |
| P1 | 本地同屏多人 | 已实现基础版 | 1-4 名预设调查员，Together/Split 两种探索方式 |
| P1 | 存档 / 读档 / 存档管理 | 已实现 | `trpg-saves-v2` 最近存档、存档列表、指定载入和删除 |
| P2 | 5 步自定义角色创建 | Backlog | 保留骰点、技能分配、导出设计 |
| P2 | 完整战斗轮序 / 伤害 / SAN 疯狂 | Backlog | 规则保留，当前 MVP 未自动化 |
| P3 | 骰子动画 / 音效 / BGM | Backlog | 当前只有基础检定卡和静态反馈 |

1. **立绘素材升级路线**——现有素材继续使用；正式版是写实、插画、AI 生成还是手绘仍待定。
2. **场景图升级路线**——`scene_main_fog_london.gif` 作为高表现力主视觉保留；其余场景最终美术制作方式待定。
3. **本地模型支持**——OpenAI / Anthropic / MiMo / 自定义端点已支持；是否投入本地模型路线仍待评估。
